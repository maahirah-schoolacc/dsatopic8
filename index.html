LESSON 1 / 4~20 min

Introduction to Stacks and Their LIFO Behaviour

Why restricting yourself to only one end of a collection turns out to be a feature, not a limitation.

Why stacks matter

In computer science, how we organise and manage data can make all the difference in solving problems efficiently. Data structures are the backbone of every program, helping us store, access, and manipulate information in powerful ways. A stack is one of the simplest of these structures — and one of the most widely used, from undo buttons to browser navigation to how function calls are tracked behind the scenes.

Structure — what is a stack?

A stack is a linear data structure that organises items in a specific order, allowing you to add and remove elements only from one end, known as the top. This follows the Last-In, First-Out (LIFO) principle: the most recently added item is always the first one removed. Items further down the stack simply cannot be accessed directly — only the top is ever visible.

Key stack terminology

TermDefinition
StackA linear data structure where items are added and removed from one end, called the top, following the LIFO principle.
LIFOLast-In, First-Out — the last item added to a stack is the first one to be removed.
PushThe operation of adding an item to the top of the stack.
PopThe operation of removing (and returning) the top item from the stack.
Peek (or Top)The operation that returns the top item of the stack without removing it.

Operation — push, pop, and peek

Push: add an item to the top. If the stack is empty and you push 'A', 'A' becomes the top; push 'B' next and 'B' sits on top of 'A'. Pop: remove the top item. If 'B' sits on top of 'A', popping removes 'B', leaving 'A' as the new top. Peek: look at the top item ('B') without removing it — both items remain in the stack afterwards.

Only the top slot (dark) is ever directly reachable — everything below it is "locked" until the items above are popped off first.

Pitfall — items below the top are never directly reachable

A common mix-up: assuming you can grab or inspect an item in the middle of the stack. You can't — push, pop, and peek all operate only at the top. To reach an item underneath, you must pop everything above it first, one at a time, in reverse order of how it was added.

Think of a stack like a stack of plates in your kitchen: you add clean plates to the top, and when you need one, you take the top plate off first. The undo feature in software works the same way — your most recent action sits on top of a stack, so it's the first thing undone. Even your browser's back button uses a stack to remember visited pages, taking you back in the exact reverse order.

Stack use cases in the real world

The tell: does the most recently added item need to come out first? Undo, back buttons, and function calls all say yes. A ticket queue is first-come-first-served (opposite order), and shuffle mode isn't ordered at all.

Quick self-check

Which of the following best describes the LIFO principle?
"The last item added is the first removed." Not first-in-first-out, not random order, and items aren't all removed at once.
A stack contains A, B, and C, with C on top. What happens when you call pop()?
C is removed and returned from the top of the stack — B becomes the new top. Pop always acts on the top item, never the bottom.
You need to check which item is on top of a stack before deciding what to do next, without disturbing the stack. Which operation do you use?
Peek — it returns the top item without removing it, leaving the stack completely unchanged.
LESSON 2 / 4~30 min

Implementing the Stack ADT in Python

From an abstract blueprint to a working, testable Python class.

From concept to code: Abstract Data Types

An Abstract Data Type (ADT) is like a blueprint for a data structure. It defines a set of values and the operations that can be performed on those values, without specifying how those operations are implemented — much like an architectural blueprint describes what rooms a house should have, but not the exact materials or construction methods. ADTs let programmers focus on what a data structure should do, before worrying about how it does it.

Structure — key terms

ADT: a model that defines a data structure by its behaviour (operations) rather than its implementation. Stack ADT: an ADT where items are added and removed from only one end, following LIFO. Interface: the set of operations an ADT provides (push, pop, peek, …). Implementation: the actual code used to realise those operations in a real language.

The Stack ADT interface

OperationWhat it does
Stack()Creates a new, empty stack.
isEmpty()Returns a boolean indicating whether the stack is empty.
length()Returns the number of items currently in the stack.
push(item)Adds the given item to the top of the stack.
pop()Removes and returns the top item — cannot be called on an empty stack. After a pop, the next item becomes the new top.
peek()Returns the top item of a non-empty stack without removing it or modifying the stack's contents.

Operation — building it in Python

We use Python's built-in list to store stack items internally, then wrap it in a class to encapsulate the stack's behaviour. Encapsulation protects the internal data — users interact with the stack only through defined methods like push, pop, and peek, never by touching the list directly.

Python implementation

# Implementation of the Stack ADT using a Python list.
class Stack:
    # Creates an empty stack.
    def __init__(self):
        self._theItems = list()

    # Returns True if the stack is empty or False otherwise.
    def isEmpty(self):
        return len(self) == 0

    # Returns the number of items in the stack.
    def __len__(self):
        return len(self._theItems)

    # Returns the top item on the stack without removing it.
    def peek(self):
        assert not self.isEmpty(), "Cannot peek at an empty stack"
        return self._theItems[-1]

    # Removes and returns the top item on the stack.
    def pop(self):
        assert not self.isEmpty(), "Cannot pop from an empty stack"
        return self._theItems.pop()

    # Push an item onto the top of the stack.
    def push(self, item):
        self._theItems.append(item)
MethodWhat it does
push(item)Appends item to the internal list — it becomes the new top.
pop()Asserts the stack isn't empty, then removes and returns self._theItems[-1] via list's own .pop().
peek()Asserts the stack isn't empty, then returns self._theItems[-1] without removing it.
isEmpty()Returns True when len(self) == 0.
__len__()Returns len(self._theItems) — lets you call len(stack) directly.

Cost — every core operation is O(1)

Because the internal list only ever grows or shrinks from its end, push, pop, and peek are all constant-time operations — no shifting, no scanning. This is exactly why a Python list (appending/popping at the end) is a natural fit for implementing a stack: the "top" of the stack maps directly onto the end of the list.

Pitfall — popping or peeking an empty stack

If you attempt to pop() or peek() an empty stack, the method should raise an error or assertion — there is no top item to return, so returning something anyway would silently corrupt your program's logic. Always call isEmpty() first to check the stack's state before popping or peeking. Assert statements enforce this precondition and catch the mistake immediately, rather than letting invalid data slip through.

A well-implemented stack is the building block for countless algorithms and applications — mastering its interface and implementation is key to programming success.

Quick self-check

Which method would you use to check if a stack is empty in the Python implementation above?
isEmpty() — returns True if the stack has no items, False otherwise. It's commonly used before calling pop() or peek() to avoid errors.
Which Python design best implements a stack using object-oriented principles?
A class that stores items in an internal list and provides methods such as push, pop, and peek — this encapsulates the data and controls access through a defined interface.
What happens if you call pop() on an empty stack, and why does it matter?
It raises an assertion error rather than returning invalid data — this prevents the program from crashing unexpectedly or silently using garbage data later on.
LESSON 3 / 4~25 min

Understanding Prefix, Infix, and Postfix Expressions

The same calculation, written three different ways — and only one of them a computer can evaluate without extra rules.

Why notation matters

When you solve math problems, you probably write expressions like 2 + 3 or (A + B) * (C - D). But computers process these expressions differently than humans do. Understanding infix, prefix, and postfix notation is essential for programming, parsing, and evaluating expressions efficiently.

Structure — three notations, one calculation

Infix places the operator between its operands — the way we normally write math, e.g. 2 + 3. Prefix places the operator before its operands, e.g. + 2 3. Postfix places the operator after its operands, e.g. 2 3 +. An operand is a value (a number or variable); an operator is a symbol such as +, -, *, /.

Same calculation, side by side

NotationFormExample — "add A and B, subtract C and D, then multiply"
Infixoperator between operands(A + B) * (C - D)
Prefixoperator before operands* + A B - C D
Postfixoperator after operandsA B + C D - *

Each notation rearranges the operators and operands differently, but all three represent exactly the same calculation.

Converting infix → postfix / prefix

  1. Identify the infix expression — e.g. A * B + C / D.
  2. Add parentheses for clarity, applying operator precedence: ((A * B) + (C / D)).
  3. Move each operator to the end of its group for postfix, or the front for prefix.
  4. Remove all parentheses to get the final form.
To postfixTo prefix
((A B *) (C D /) +)(+ (* A B) (/ C D))
→ A B * C D / +→ + * A B / C D

Operation — why prefix and postfix are preferred in computing

Prefix and postfix notations let computers evaluate expressions without worrying about parentheses or operator precedence rules. With postfix in particular, a stack can process the expression efficiently in a single left-to-right pass — which is exactly what Lesson 4 covers.

Strengths and weaknesses of each notation

NotationStrengthsWeaknesses
InfixFamiliar and easy for humans to read/write — matches how we naturally express maths.Ambiguous for computers; needs parentheses and precedence rules to parse correctly.
PrefixRemoves ambiguity — operator always comes first, no parentheses needed, simplifies parsing.Less intuitive for people, since it doesn't match how we usually write maths.
PostfixIdeal for stack-based evaluation — no parentheses or precedence rules needed; used by compilers and calculators.Less familiar to humans; harder to interpret at a glance.

Cost — the trade-off in one line

Infix wins on human readability. Prefix and postfix win on computational simplicity, because the structure of the expression alone (not extra rules) determines the order of evaluation — which is precisely the ambiguity infix has to resolve with parentheses and precedence.

Pitfall — don't just reverse the operators

It's tempting to think postfix is "infix with the operator moved to the end," full stop. But operand order also matters, especially for non-commutative operators like - and /. A B + C D - * means (A + B) * (C - D) — the grouping is preserved, just rearranged; it does not mean (B + A) * (D - C).

Quick self-check

Which of the following is the correct postfix form of the infix expression A * (B + C) / D?
A B C + * D / — B and C combine first (matching the parentheses), then the result multiplies with A, then the whole thing divides by D.
Why don't prefix and postfix expressions need parentheses?
Because the position of each operator relative to its operands fully determines the structure and order of evaluation — there's no ambiguity left for parentheses to resolve.
LESSON 4 / 4~25 min

Evaluating Postfix Expressions Using Stacks

How a calculator turns a row of tokens into a single number, in one left-to-right pass.

Key terms in postfix evaluation

TermDefinition
OperandA value or variable used in a mathematical operation.
OperatorA symbol such as +, -, *, / that tells the computer what operation to perform.
StackThe data structure that stores operands during evaluation, following LIFO.
TokenEach individual part of the expression — either an operand or an operator.
ResultThe final value left on the stack after the entire expression is evaluated.

Structure — the postfix evaluation algorithm

Scan the expression from left to right, one token at a time. Operands get pushed. Operators trigger a pop-compute-push cycle. By the time you reach the end, the stack holds exactly one value: the answer. No parentheses, no precedence table required — the stack does all the bookkeeping.

  1. Start scanning — begin at the leftmost token, move right one token at a time.
  2. Push operands — if the token is an operand, push its value onto the stack.
  3. Pop for operators — if the token is an operator, pop the top two values off the stack.
  4. Apply the operation — the first value popped is the right operand; the second value popped is the left operand.
  5. Push the result back onto the stack, ready for later operators.
  6. Final result — once every token is processed, the one value left on the stack is the answer.

Pitfall — operand order is not symmetric

For a token stream ... x y op, y is popped first (the right operand) and x is popped second (the left operand), and the operation applied is x op y — not y op x. This doesn't matter for + or *, but it absolutely matters for - and /: mixing up the order silently gives you the wrong answer, with no error raised.

Error — too many values left on the stack

A valid postfix expression must leave exactly one final value on the stack. This example finishes with 16 and 7 both still sitting there — meaning not every operand was consumed by an operator. That's the signature of an invalid (malformed) postfix expression.

Postfix evaluation: common questions

What if there aren't enough operands on the stack when an operator appears?
The expression is invalid — this usually signals a mistake in the expression or a missing operand.
How are multi-digit numbers or variable names handled?
They're treated as single tokens. As long as tokens are clearly separated (e.g. by spaces), there's no confusion between a number like 12 and two separate operands 1 and 2.
How does operator precedence affect postfix evaluation?
It doesn't need to — precedence is already baked into the order of the tokens. The stack just processes them left to right; there are no precedence rules to apply during evaluation itself. This is one of postfix's main advantages over infix.

Cost — one pass, O(n)

Each token is read exactly once, and each operator triggers exactly one pop-pop-push cycle (all O(1) stack operations). Evaluating a postfix expression with n tokens is therefore O(n) overall — a single linear scan, with no backtracking and no re-parsing needed.

Quick self-check

What will be the result of evaluating the postfix expression: 5 2 3 * + 4 - ?
7. Push 5, 2, 3. *: pop 3,2 → 2*3=6 → stack [5,6]. +: pop 6,5 → 5+6=11 → stack [11]. Push 4 → [11,4]. -: pop 4,11 → 11-4=7 → stack [7].
CRAM SHEET~10 min — last pass before practice

One-Page Recap

If you only remember these, remember these.

1. What a stack is

A linear structure where items are added and removed only from the top, following LIFO (Last-In, First-Out). Items below the top are never directly reachable.

2. The three core operations

Push — add to the top. Pop — remove and return the top item. Peek — return the top item without removing it. All three (plus isEmpty and length) make up the Stack ADT interface.

3. Python implementation & cost

Wrap a Python list in a class (self._theItems); push = append, pop = list's own .pop(), peek = [-1]. Because everything happens at the end of the list, every core operation is O(1).

4. The classic error

Calling pop() or peek() on an empty stack should raise an assertion/error — always check isEmpty() first.

5. Infix vs. prefix vs. postfix

Infix (operator between operands) — natural for humans, needs parentheses/precedence. Prefix (operator before) and postfix (operator after) — remove ambiguity, no parentheses needed; postfix is what stacks evaluate directly.

6. Evaluating postfix with a stack

Scan left to right. Operand → push. Operator → pop y first (right operand), then x (left operand), compute x op y, push the result. One value left at the end = the answer; anything else = an invalid expression.
